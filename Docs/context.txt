CodeForge Context

1. Project Overview

CodeForge is a Java-based desktop IDE built with JavaFX.
It currently focuses on Python editing, file management, execution, diagnostics, and lightweight runtime summaries.
The application combines:

- a RichTextFX-based code editor
- multi-tab document sessions
- workspace browsing and file actions
- multiple terminal tabs
- parser-backed diagnostics
- AI Snapshot summaries
- external Python execution through `PythonRunner`

The internal compiler package is already present and currently powers validation more than full execution.

2. Entry Point

`Launcher.java` forwards directly to `MainApp.main(args)`.
`MainApp.main(args)` loads configuration through `ConfigManager.load()` and then launches the JavaFX application.
`MainApp.start(Stage stage)` builds the visible shell:

- toolbar
- workspace explorer
- editor area
- bottom tabs for terminals, problems, and AI Snapshot
- status bar

Most interaction logic is delegated to `WorkbenchController`.

3. Editor System

`CodeEditor.java`

- wraps a RichTextFX `CodeArea`
- stores the live editor text
- exposes `getCode()` and `setCode(...)`
- tracks dirty state
- exposes content and dirty listeners
- supports `moveTo(line, column)` for diagnostics navigation

`EditorTabSession.java`

- represents one open document session
- owns a `Tab`
- owns one `CodeEditor`
- tracks the current `Path`, if any
- stores diagnostics for that session
- stores the current AI Snapshot result for that session

`EditorTabManager.java`

- owns the editor `TabPane`
- creates untitled scratch sessions
- opens file-backed sessions from disk
- saves sessions
- saves sessions directly to a specified path
- tracks the active session
- keeps one session per normalized file path

`EditorController.java`

- runs the active editor session
- asks `LanguageService` for a process
- attaches that process to a terminal
- forwards run lifecycle events through `ExecutionListener`

How files are opened

- from the file chooser through `WorkbenchController.openFileChooser()`
- from the workspace tree through `WorkspaceView`
- from direct file creation through `WorkbenchController.createOrOpenFile(...)`

How content is stored

- the authoritative in-memory text is in `CodeEditor`
- `EditorTabSession` stores metadata and the editor instance, not a separate text copy

How text is accessed programmatically

- `EditorTabManager.getActiveSession()`
- `EditorTabSession.getEditor()`
- `CodeEditor.getCode()`

4. File Management

`FileManager.java` is the low-level filesystem utility.
It currently handles:

- file open chooser
- save chooser
- file reads
- file writes
- ensure-file-exists behavior
- rename
- delete
- copy

Files are represented as `java.nio.file.Path`.
`EditorTabManager` stores open file-backed sessions by normalized path so the same file is not opened twice in separate tabs.

5. Terminal System

`TerminalManager.java`

- owns the terminal `TabPane`
- creates and selects terminal sessions
- returns the active terminal session

`TerminalSession.java`

- owns one terminal tab and view
- accepts an externally created `Process`
- streams stdout into the UI
- stores a transcript in memory
- forwards events through `TerminalSessionListener`
- now exposes `getTranscript()` for safe transcript retrieval

`TerminalView.java`

- displays terminal output in a read-only `TextArea`
- provides a text field and send button for stdin
- appends output through `print(...)`

How commands are executed

The terminal layer does not start arbitrary shell commands by itself.
It displays and manages a `Process` created elsewhere, usually by `PythonRunner`.

How output is captured

`TerminalSession.streamOutput()` reads from the process input stream, appends chunks to the transcript, updates the terminal UI, and forwards the latest full transcript to listeners.

How output can be retrieved

- through `TerminalSessionListener` callbacks while a run is active
- through `TerminalSession.getTranscript()` for the current active terminal session

6. Python Execution

`PythonRunner.java` is the current runtime path for Python execution.

It:

- writes the active editor content to the backing file when a file path exists
- creates a temporary `.py` file when the session is still untitled
- starts `python <file-path>`
- merges stderr into stdout with `redirectErrorStream(true)`

That process is then attached to `TerminalSession`, so runtime output and errors appear in the terminal transcript.

7. Language / Analysis System

`LanguageService.java`

- validates source text through the internal lexer and parser
- turns lexer and parser exceptions into `Diagnostic` objects
- delegates execution to `PythonRunner`

`ProblemsView.java`

- displays the diagnostics of the active editor session
- lets the user double-click a diagnostic to navigate the caret to that location

How errors are detected

- pre-run syntax and parser errors are detected by `LanguageService.validate(...)`
- runtime errors are not shown in the Problems panel
- runtime output and runtime failures appear in the terminal and AI Snapshot flow

8. AI Snapshot System

`AISnapshotService.java`

- summarizes pre-run validation state
- analyzes terminal transcript plus exit code after execution
- uses local fast-path checks first for common Python failures
- can fall back to a model-backed provider through `LLMClient`
- returns structured `AISnapshotResult` values

`AISnapshotView.java`

- renders four labeled values
- Problem
- Cause
- Fix
- Confidence

What data it receives

- active session diagnostics before execution
- terminal transcript and exit code after execution

How it analyzes terminal output

- trims and truncates long transcript text
- checks common runtime patterns such as `ModuleNotFoundError`, `SyntaxError`, `NameError`, `FileNotFoundError`, and `ImportError`
- falls back to the configured provider only when local detection does not already explain the output

9. Internal Flow

User writes code -> runs code -> output appears

1. The user edits text inside `CodeEditor`.
2. `WorkbenchController` validates the active session through `LanguageService`.
3. Diagnostics are stored in `EditorTabSession`.
4. `ProblemsView` is refreshed for the active session.
5. `AISnapshotService.summarizeValidation(...)` updates the session snapshot.
6. `WorkbenchController.runActive()` triggers `EditorController.runActiveFile()`.
7. `LanguageService.runExternally(...)` delegates to `PythonRunner`.
8. `PythonRunner` writes the current code to disk and starts the Python process.
9. `TerminalSession` streams output into `TerminalView` and stores the transcript.
10. When execution completes, `WorkbenchController` sends the transcript to `AISnapshotService`.
11. `AISnapshotView` updates with the latest structured result for the active session.

10. Current File Creation Workflow

CodeForge now supports both temporary and file-backed work.

Scratch workflow

- `New Scratch` creates an untitled tab
- the content stays in memory until the user saves it
- `PythonRunner` can still execute untitled work by using a temporary file

Named file workflow

- `Create File` creates or opens a real file on disk immediately
- when a workspace root is open, that root is the preferred location
- otherwise the active file's parent directory is used when available
- if no clean location is available, the app falls back to a save dialog

Save behavior

- file-backed sessions save directly to their current path
- untitled sessions prompt for a real path
- saving to a specific path updates session metadata and tab title through `EditorTabManager`

11. Current Supporting Services

`ConfigManager.java`

- loads `.env` once at startup
- serves configuration values such as `GROQ_API_KEY`

`AISnapshotResult.java`

- keeps the snapshot output structured and session-friendly

`ExecutionListener.java` and `TerminalSessionListener.java`

- keep editor-run orchestration and terminal process events loosely connected

Current project character

CodeForge is currently strongest as a structured desktop IDE shell with:

- editor sessions
- file-backed and scratch workflows
- Python execution
- diagnostic feedback
- concise runtime summaries

The major responsibilities of the codebase now have a clear home, which makes the project easier to extend without rewriting the whole shell.
